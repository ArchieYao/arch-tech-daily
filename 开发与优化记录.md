# 建筑科技日报 — 开发与优化记录

本文档汇总基于开源项目 ai-daily 与 we-mp-rss 所做的产品改造、功能开发与问题修复。

---

## 一、产品与架构改造（首轮落地）

### 1.1 产品定位与前端结构

- **目标**：从「全球技术/AI 精选」改为「前沿 AI 资讯 + 前沿建筑科技资讯」双板块日报，面向建筑行业从业者。
- **前端**：由单页「最新一期 + 设置弹窗」改为三视图 SPA：
  - **历史报告列表**（`#/reports`）：日期 + 报告标题（≤25 字），点击进入详情。
  - **报告详情**（`#/reports/:date`）：标题、时间、核心要点、详细资讯（分 AI 资讯 / 建筑科技资讯）、资讯来源、PDF 下载、返回列表。
  - **管理后台**（`#/admin`）：独立入口，需登录后使用。
- **涉及文件**：`public/index.html`、`public/app.js`、`public/style.css`、`server.mjs`（SPA 回退路由）。

### 1.2 报告标题与列表数据

- **数据**：`lib/db.mjs` 中 digest 增加 `reportTitle` 字段，`getDigestList` 返回 `report_title`。
- **生成**：`lib/highlights.mjs` 新增 `generateReportTitle(articles, ...)`，根据入选文章/看点用 AI 生成 ≤25 字中文标题；`server.mjs` 的 `runDigestGeneration` 在生成 highlights 后调用并写入。
- **涉及文件**：`lib/db.mjs`、`lib/highlights.mjs`、`server.mjs`。

### 1.3 RSS 双板块与默认源

- **默认源**：清空原 90 个技术源，改为「AI 板块」+「建筑科技板块」；建筑科技默认使用 9 个微信公众号 RSS（来自 we-mp-rss 导出）。
- **数据**：RSS 源增加 `domain` 标签（`ai` / `building`），抓取时携带到文章 `sourceDomain`，写入 digest 的 `source_domain`。
- **涉及文件**：`lib/rss-list.mjs`（`RSS_FEEDS`、`RSS_FEEDS_BUILDING`）、`lib/feeds.mjs`、`server.mjs`（POST `/api/rss-sources` 校验 domain、生成流程按 domain 分池）。

### 1.4 筛选与入选逻辑

- **每源最多 1 篇**：同一 RSS 源最终最多入选 1 篇，取该源内评分最高的一篇，其余由其他源顶替。
- **双板块数量**：前 10 条为 AI 资讯（`sourceDomain !== 'building'`），后 5 条为建筑科技（`sourceDomain === 'building'`）。
- **涉及文件**：`server.mjs`（`runDigestGeneration` 内 `aiCandidates` / `buildingCandidates`、`takePerSource`）、`lib/scoring.mjs`（评分 prompt 区分 AI/建筑科技相关性）。

### 1.5 AI 流程与文案

- **评分**：`lib/scoring.mjs` 评分维度与分类面向「前沿 AI + 建筑科技」，分类标签含 `ai-ml`、`building-tech` 等；按源 `domain` 分池后分别排序取 top。
- **摘要**：`lib/summarize.mjs` 角色与说明改为「建筑科技日报」、面向建筑与 AI 受众。
- **看点与标题**：`lib/highlights.mjs` 今日看点按两大板块归纳；报告标题优先「多来源共同热点」或首条标题提炼，≤25 字、准确专业。
- **涉及文件**：`lib/scoring.mjs`、`lib/summarize.mjs`、`lib/highlights.mjs`。

## 二、PDF 与下载

### 2.1 PDF 生成

- **实现**：`server.mjs` 新增 `GET /api/digest/:date/pdf`，使用 `pdfkit` 从报告数据生成 PDF。
- **中文乱码修复**：通过环境变量 `PDF_FONT_PATH` 指定中文字体（如 `NotoSansSC-VariableFont_wght.ttf`），容器内挂载 `public/fonts`。
- **涉及文件**：`server.mjs`、`docker-compose.yml`（`PDF_FONT_PATH`）、`public/fonts/`。

### 2.2 下载行为

- **需求**：详情页「下载 PDF」直接下载文件，而非打开打印页。
- **实现**：前端「下载 PDF」按钮改为打开 `GET /api/digest/:date/pdf`（`Content-Disposition: attachment`），触发浏览器下载。
- **涉及文件**：`public/app.js`（`downloadPdfBtn` 逻辑）。

### 2.3 打印页

- **路由**：`GET /api/digest/:date/print` 输出打印友好 HTML，按 `source_domain` 分「AI 资讯」「建筑科技资讯」两段展示；与 PDF 内容结构一致。
- **涉及文件**：`server.mjs`。

## 三、we-mp-rss 集成

### 3.1 客户端与代理

- **新建**：`lib/werss-client.mjs`，封装 we-mp-rss 的 HTTP 调用（login、getQrCode、getQrImage、getQrStatus、searchMp、subscribeMp、getMpList、getRssUrl）。
- **服务端**：`server.mjs` 中通过环境变量 `WERSS_BASE_URL`、`WERSS_USERNAME`、`WERSS_PASSWORD` 连接 we-mp-rss，JWT 缓存后代理部分 API；管理类接口需 admin token。
- **涉及文件**：`lib/werss-client.mjs`、`server.mjs`（`/api/werss/status`、`login`、`qr/code`、`qr/image`、`qr/status`、`search`、`subscribe`、`mps`、`export`）。

### 3.2 一键导出公众号 RSS 清单

- **实现**：不依赖 we-mp-rss 的 HTTP 列表接口，改为 ai-daily 后端直接读 we-mp-rss 的 SQLite 库（`db.db`），通过 volume 挂载 `we-mp-rss/data` 到 ai-daily-web 容器，使用 `sql.js` 查询 `feeds` 表，生成「公众号名称 + RSS URL」列表并返回 JSON。
- **前端**：管理后台 RSS 订阅源区块增加「导出公众号 RSS 清单」按钮，请求 `GET /api/werss/export`，展示列表并支持「全部添加到 RSS 源」或单条添加（添加时 `domain: 'building'`）。
- **涉及文件**：`server.mjs`（`/api/werss/export`、`WERSS_DB_PATH`）、`public/index.html`、`public/app.js`、`docker-compose.yml`（volume `./we-mp-rss/data:/app/we-mp-rss/data:ro`）。

### 3.3 公众号管理入口与说明

- **管理后台**：新增「公众号管理」tab，说明通过 we-mp-rss 管理订阅，并提示 we-mp-rss 默认账号（admin / admin@123），按钮新窗口打开 we-mp-rss 地址。
- **涉及文件**：`public/index.html`、`public/app.js`。

### 3.4 手动刷新（已移除）

- 曾增加「手动一键刷新 we-mp-rss 所有公众号最新文章」按钮，调用 `/api/werss/refresh` 触发 we-mp-rss 消息任务 run；因执行报错（Method Not Allowed 等）且用户不需要，已从管理后台移除该按钮及相关前端逻辑；后端路由可保留或后续清理。
- **涉及文件**：`public/index.html`、`public/app.js`（移除按钮与 `refreshWxMpArticles` 等）。

## 四、统一管理后台认证

### 4.1 认证体系改造

- **原状**：全站使用 `SITE_PASSWORD`，登录后访问所有页面。
- **目标**：日报首页、历史报告、报告详情、分享页公开访问；仅管理后台需用户名+密码登录（默认 archie / archie2077），支持修改密码；we-mp-rss 保留独立登录，管理后台仅提供入口与说明。

### 4.2 后端

- **lib/auth.mjs**：重写为多用户体系；存储 `{ users: [{ username, salt, hash }] }`；首次启动或无用户时 `initDefaultAdmin()` 创建默认管理员（可被 `ADMIN_USERNAME` / `ADMIN_PASSWORD` 覆盖）；新增 `verifyUser(username, password, ip)`、`changePassword(username, oldPassword, newPassword)`、`getSessionUser(token)`；session 含 `username`；删除 `setPassword`、`isPasswordSet`；`authMiddleware` 仅对管理 API 校验 token（公开 API 如 `/api/digest/*`、`/api/digests`、`/api/stats`、`/api/auth/login`、`/api/auth/status` 等放行）。
- **server.mjs**：移除 `SITE_PASSWORD` 及自动设密逻辑；启动时调用 `initDefaultAdmin()`；`POST /api/auth/login` 改为接收 `{ username, password }`；新增 `GET /api/auth/me`、`POST /api/auth/change-password`；管理 API 需带 `X-Auth-Token`。
- **涉及文件**：`lib/auth.mjs`、`server.mjs`。

### 4.3 前端

- **登录**：移除全局 `#loginScreen`；在 `#adminPage` 内增加 `#adminLoginBlock`（用户名、密码、登录按钮），登录成功后显示 `#adminMainBlock`（含退出、已登录用户名）。
- **Token**：前端使用 `localStorage.admin_token`，请求管理 API 时在 Header 中附带。
- **流程**：应用启动不再全站 `checkAuth()`，直接进入首页；进入 `#/admin` 时先请求 `/api/auth/me`，无效则显示管理员登录表单，有效则显示管理面板。
- **新增**：「公众号管理」tab、「修改密码」区域（旧密码、新密码、确认、保存）；修改密码调用 `POST /api/auth/change-password`。
- **涉及文件**：`public/index.html`、`public/app.js`。

### 4.4 环境与部署

- **清理**：`docker-compose.yml`、`.env.example` 中移除 `SITE_PASSWORD`；`.env.example` 可增加 `ADMIN_USERNAME` / `ADMIN_PASSWORD` 说明。
- **涉及文件**：`docker-compose.yml`、`.env.example`。

## 六、Bug 修复

### 6.1 详情页 / PDF 文章错位

- **现象**：详情页或 PDF 中，公众号名称、标题、摘要与链接不对应（如 AI 第一条显示建筑公众号内容）。
- **原因**：摘要/评分批量调用 AI 时使用全局文章下标，AI 返回按 batch 内 0-based 编号，写回时未映射到原始索引导致覆盖错位。
- **修复**：`server.mjs` 传入 `summarizeArticles` / `scoreArticles` 的为按入选列表的 0-based 索引；`lib/summarize.mjs`、`lib/scoring.mjs` 内部按 batch 处理时用 `batch[localIdx].index` 映射回全局索引再写入 `summaries`/`scores`，保证每条与 `title_zh`、`summary`、`link` 一一对应。
- **涉及文件**：`server.mjs`、`lib/summarize.mjs`、`lib/scoring.mjs`。

### 6.2 报告详情只显示 AI 资讯

- **现象**：详情页仅展示 AI 资讯，建筑科技区块为空或错位。
- **修复**：前后端统一按 `source_domain === 'building'` 拆分「建筑科技资讯」，其余为「AI 资讯」；详情页、PDF、打印页均按此分段渲染并连续编号。
- **涉及文件**：`server.mjs`（digest 数据与 PDF/print 分段）、`public/app.js`（`showReportDetail` 中 `aiArticles` / `buildingArticles`）。

### 6.3 PDF 中文乱码

- **现象**：生成的 PDF 中文为乱码。
- **修复**：使用支持中文的字体文件，通过 `PDF_FONT_PATH` 指定路径，在 `pdfkit` 中设置 `doc.font(PDF_FONT_PATH)`；字体文件置于 `public/fonts/` 并在 Docker 中挂载。
- **涉及文件**：`server.mjs`、`docker-compose.yml`、`public/fonts/`。

### 6.4 导出 RSS 清单 401 / Failed to fetch

- **现象**：管理后台点击「导出公众号 RSS 清单」报 Failed to fetch。
- **原因**：`/api/werss/export` 属管理 API，需带 `X-Auth-Token`；使用新域名或新设备访问时未登录，无 token 导致 401，前端 catch 显示为 Failed to fetch。
- **处理**：确保在管理后台登录后再点击导出；或在浏览器该域名下重新登录（archie / archie2077）。

### 6.5 端口与访问方式

- **现象**：`curl http://127.0.0.1:3456` 连接被拒。
- **原因**：`docker-compose` 将 ai-daily-web 映射为 `0.0.0.0:80->3456`，宿主机只监听 80，不监听 3456。
- **结论**：通过公网 IP 访问时使用 `http://<公网IP>/`（80 端口），无需带 3456。

## 十、内存泄漏修复与服务器稳定性优化（2026-03-23）

### 10.1 背景

- **环境**：腾讯云轻量服务器，2GB 内存，运行 ai-daily + we-mp-rss 两个服务。
- **现象**：稳定运行约 5 天后内存占用超过 90%，服务器瘫痪被迫重启。
- **触发条件**：we-mp-rss 后台配置了每 5 分钟刷新公众号消息的定时任务，加剧了内存增长。

### 10.2 根因分析

| 问题 | 严重度 | 说明 |
|------|--------|------|
| dbCache 无限增长 | 高 | `lib/db.mjs` 将整个 `digests.json` 常驻内存，每天新增日报但从不清理旧数据的 articles 数组 |
| 翻译缓存只在启动时清理 | 中 | `db.translations` 仅在进程启动时 `pruneTranslations(30)`，长期运行后持续膨胀 |
| sql.js WASM 重复加载 | 中 | `/api/werss/export` 每次请求都 `import('sql.js')` 并 `initSqlJs()`，加载 ~10-20MB WASM 模块 |
| SSE 死连接无超时 | 中 | `sseClients` Set 仅在客户端主动断开时清理，僵尸连接永久占用内存 |
| Node.js 无内存上限 | 低 | 默认 V8 堆无上限，会耗尽所有可用内存后才触发 OOM |

### 10.3 修复措施

#### A. dbCache 自动清理旧日报（`lib/db.mjs`）
- 新增 `pruneOldDigests(keepDays)` 函数：清理超过 30 天的日报 `articles` 数组（保留日期、标题等元信息，释放大体积文章数据）。
- 启动时立即执行一次清理。
- 每 6 小时自动执行清理（旧日报 + 翻译缓存），防止长期运行导致内存膨胀。

#### B. sql.js WASM 实例缓存（`server.mjs`）
- 新增 `getSqlJs()` 函数缓存 `initSqlJs()` 返回的 SQL 实例，避免每次 `/api/werss/export` 请求都重复加载 WASM 模块。

#### C. SSE 连接超时保护（`server.mjs`）
- `sseClients` 从存储 `res` 改为存储 `{ res, connectedAt }` 对象。
- 新增 30 分钟超时机制：每 60 秒检查一次，超时连接自动断开并从 Set 中移除。

#### D. Node.js 内存限制与监控（`package.json` + `server.mjs`）
- 启动参数新增 `--max-old-space-size=768`（限制 V8 堆最大 768MB，为系统和 we-mp-rss 留空间）。
- 启动参数新增 `--expose-gc`（允许代码主动触发 GC）。
- 新增每 10 分钟内存监控日志，输出 RSS、Heap、SSE 连接数、限流条目数。
- 当 RSS 超过 1.2GB 时自动触发 `global.gc()`。

### 10.4 建议的运维调整

- **we-mp-rss 消息任务刷新频率**：从每 5 分钟改为每 2 小时，或完全依赖 ai-daily 内置的凌晨 0:00 自动刷新（`server.mjs` 第 496 行）。
- **部署后监控**：观察日志中 `[memory]` 输出，确认 RSS 稳定不持续上涨。

### 10.5 涉及文件

| 文件 | 变更 |
|------|------|
| `lib/db.mjs` | 新增 `pruneOldDigests()`；新增每 6 小时定期清理定时器 |
| `server.mjs` | 新增内存监控定时器；启动时调用 `pruneOldDigests(30)`；缓存 sql.js 实例；SSE 超时保护 |

## 十一、首页骨架屏（刷新占位）误显示修复（2026-03-02）

### 11.1 现象

- 打开首页（默认 `#/reports` 历史报告列表）时，页面上方出现多块**骨架屏/占位块**，真正的「历史报告」列表被压在下方。
- 用户期望：首屏直接展示历史报告列表，不要这些「刷新中」占位模块。

### 11.2 根因

| 原因 | 说明 |
|------|------|
| **A. MutationObserver 与路由竞态** | `public/enhancements.js` 中 `initSkeleton` 曾用 `MutationObserver` 在 `#mainApp` 显示后自动去掉 `#skeletonView` 的 `hidden`；而 `app.js` 的 `route()` → `showView('list')` 会先隐藏骨架。观察者回调晚于同步的 `route()`，会再次显示骨架。 |
| **B. 列表路由无 `digestRendered`** | 默认历史列表不会调用 `renderDigest()`，**不会**派发 `digestRendered`，骨架无法靠该事件自动关闭。 |
| **C. Tailwind `hidden` 与 `flex` 同元素冲突** | `#skeletonView` 曾写 `class="hidden flex flex-col"`，CDN 生成的 `.flex` 可能覆盖 `.hidden` 的 `display:none`，骨架**始终可见**。 |
| **D. 样式兜底** | 在 `style.css` 中为 `#skeletonView` 增加默认 `display: none !important`，与已有 `#skeletonView:not(.hidden) { display: flex !important; }` 配合，避免工具类覆盖隐藏状态。 |

### 11.3 修复措施

- **`public/enhancements.js`**：移除「随 `mainApp` 显示而自动展示骨架」的 `MutationObserver` 与 `maybeShow`；保留 `digestRendered` / `emptyStateShown` 监听，仅在需要时强制将骨架设为隐藏。
- **`public/index.html`**：`#skeletonView` 仅保留 `class="hidden"`，去掉与 `hidden` 冲突的 `flex flex-col`（展开时的 flex 由 `style.css` 的 `:not(.hidden)` 规则负责）。
- **`public/style.css`**：`#skeletonView` 增加 `display: none !important` 作为默认隐藏，与 `:not(.hidden)` 高特异性规则并存。

### 11.4 腾讯云服务器部署更新（Docker）

在服务器上进入项目目录（示例 `~/arch-tech-daily` 或 `~/ai-daily`，以实际为准），拉取最新代码后重建并启动：

